## 二、产品定义

### 2.1 一句话定位

本产品是一个**学习状态感知与分析系统**：通过连续采集学习过程中的行为信号，识别用户当前所处的学习状态与节奏，帮助低效自学者更早发现偏航、卡顿与失衡，并给出可执行的反馈建议。

### 2.2 产品边界

- 长期愿景：支持多学科学习场景的通用学习状态建模与分析
- 首个落地：编程学习场景，尤其是算法题练习、概念理解与新领域自学
- 当前不追求：通用题库平台、重型在线判题平台、全功能 AI 教练替代品

### 2.3 核心价值主张

相较于仅依赖“结果正确/错误”的学习平台，本系统重点关注：

- 用户是在**有效思考**还是已经**偏离正确路径**
- 用户是否已经进入**长期卡顿**但尚未得到提醒
- 系统能否在提交结果之前，基于过程信号给出**更早反馈**
- 学习过程能否被沉淀为**可复盘、可解释、可对比**的数据轨迹

## 三、目标用户与使用场景

### 3.1 目标用户

本产品优先服务以下用户：

- 低效自学者：难以管控学习状态、节奏与专注度的人
- 新领域初学者：有学习意愿与初步想法，但实现过程低效、容易走偏的人
- 已尝试使用 AI 辅助学习但效果不佳的人：能提问，但提问低效、反馈利用不足

### 3.2 用户典型诉求

目标用户更关心的不是“题做出来没有”，而是：

- 我现在是在真正理解，还是只是机械试错？
- 这个知识点在学习路线中的位置是什么？
- 我的学习速度和节奏是否合理？
- 我是不是已经卡住太久，需要调整策略？

### 3.3 典型场景

1. 用户启动本地应用，在浏览器中进入编程学习工作台
2. 用户选择题目或学习任务，在编辑区持续编写、修改、运行代码
3. 系统在本地持续采集行为并生成状态向量与状态轨迹
4. 执行页实时展示简洁的状态反馈与提示
5. 用户完成一次 session 后进入复盘页，查看状态变化、关键拐点与建议

## 四、问题定义与设计目标

### 4.1 当前核心问题

按优先级排序，首版需要解决的问题为：

1. 无法区分用户是在认真思考，还是已经走偏
2. 卡住太久却没有及时提醒
3. 反馈过晚，通常只有提交后才知道问题
4. 学习过程缺少复盘依据

### 4.2 产品目标

首版目标不是“自动教会用户”，而是先完成以下闭环：

- 采集学习过程行为
- 建立状态模型并完成初步分类
- 形成可解释的状态轨迹
- 在高置信情境下给出轻量提醒
- 在 session 结束后生成结构化复盘反馈

### 4.3 非目标

首版不承诺：

- 离线大模型分析能力
- 跨学科即插即用效果
- 复杂个性化课程推荐
- 高精度心理/情绪诊断
- 以 AI 替代用户自主学习过程

## 七、系统形态与交互设计

### 7.1 双模式产品结构

首版采用双模式：

- **执行页**：学习任务进行中的实时状态感知
- **复盘页**：单次 session 结束后的关键反馈与轨迹分析

### 7.2 主界面形态

执行页采用**类 IDE 工作台**布局，强调低打扰与友好呈现：

- 左侧 / 中央：任务说明、知识上下文、编辑器与运行区
- 右侧：简洁状态分析面板
- 右侧反馈不过载，只展示当前最关键状态、置信度与轻提示

### 7.3 首屏原则

用户打开后第一眼应看到：

- 当前正在做什么
- 当前处于什么学习状态
- 是否需要马上调整

### 7.4 复盘页核心内容

复盘页聚焦“反馈闭环”，展示：

- 状态时间轴
- 关键状态转折点
- 高频行为模式
- 关键指标摘要
- 2-3 条可执行建议

## 八、方法论与状态建模

### 8.1 建模原则

首版强调：

- 可解释
- 可落地
- 可逐步演进

系统不采用单一粗粒度标签，而采用**多维状态空间 + 展示层聚合映射**的设计。

### 8.2 理论状态空间（研究层）

首版保留如下多维状态结构，作为理论建模框架，并允许后续根据实验结果调整：

#### 维度 1：认知稳定性

- 连贯推进（coherent flow）
- 局部波动（micro fluctuation）
- 结构破裂（structural break）
- 完全崩塌（collapse）

#### 维度 2：探索性

- 路径锁定（path locked）
- 微调搜索（local search）
- 分支探索（branching exploration）
- 无方向试探（random probing）

#### 维度 3：阻滞 / 摩擦

- 顺滑（smooth）
- 轻微卡顿（minor friction）
- 反复回退（backtracking loop）
- 高摩擦停滞（high resistance stall）

#### 维度 4：时间节奏

- 稳定节奏（steady rhythm）
- 加速段（burst acceleration）
- 碎片化（fragmented）
- 长停滞（prolonged idle）

#### 维度 5：策略一致性

- 策略一致（consistent strategy）
- 策略微调（incremental shift）
- 策略切换（strategy switch）
- 策略迷失（strategy drift）

#### 维度 6：情绪 / 认知负荷

- 低负荷（calm）
- 集中紧张（focused tension）
- 挫败积累（frustration build-up）
- 认知过载（overload）

### 8.3 首版启用维度（工程层）

考虑首版验证难度与可解释性，工程上优先启用 3-4 个维度：

- `stability`（稳定性）
- `exploration`（探索性）
- `friction`（阻滞/摩擦）
- `rhythm`（时间节奏）

其余维度先保留为潜在变量，不作为首版对外决策依据。

### 8.4 展示层状态分组（产品层）

面向用户展示时，采用更直观的聚合状态：

**推进类**

- 稳定推进（steady flow）
- 高效推进（efficient flow）
- 谨慎推进（careful progression）

**探索类**

- 局部调试（local debugging）
- 分支探索（branch exploration）
- 试探性修改（probing edits）

**阻滞类**

- 轻度卡顿（minor stuck）
- 反复回退（looping）
- 结构性卡住（structural stuck）

**混乱类**

- 高频试错（chaotic trial）
- 无效尝试（ineffective attempts）
- 策略漂移（strategy drift）

**注意力类**

- 短暂停顿（short pause）
- 深度思考（deep thinking）
- 注意力下降（attention drop）

### 8.5 状态表示方式

状态不是互斥单标签，而是**多标签加权并存**：

- 底层以 `StateVector` 表示各维度、各状态分量及其权重
- 展示层输出 `TOP-2 状态 + 置信度`
- 自动干预仅在高置信区间触发，其余以建议形式呈现

### 8.6 方法路线

首版不直接实现完整的连续时间复杂模型，而采用轻量近似：

- 基于指数衰减的时间影响建模
- 基于滑动窗口的状态特征提取
- 基于状态平滑 / 滞后机制的状态稳定输出

理论依据与后续演进方向包括：

- Hawkes Process
- 吸引子动力系统（Attractor Dynamics）
- Langevin Equation

首版系统采用**基于指数衰减与状态平滑的轻量近似**，实现对时间依赖与状态演化的可解释建模；后续版本逐步引入参数学习与连续时间模型。

## 九、核心功能设计

### 9.1 行为采集

首版在编程学习场景下采集如下行为：

- 编辑事件（插入、删除、替换）
- 光标移动与停留
- 复制 / 粘贴事件
- 运行 / 提交事件
- session 起止与切分事件

原则：

- 仅采集行为信号，不上传原始代码内容
- 原始行为数据仅保存在本地

### 9.2 特征提取

系统以时间窗口方式提取特征，包括但不限于：

- 删除率
- 编辑间隔波动
- 停顿占比
- 回退循环程度
- 尝试频率
- 分支切换程度

### 9.3 状态推理

状态推理在本地完成，输出：

- 当前状态向量
- 聚合展示状态
- 置信度
- 触发依据

### 9.4 实时提醒 / 干预

干预原则：

- 仅在高置信度条件下触发自动提醒
- 建议性提示优先，不强打断
- 优先提示“是否需要调整策略”，而非直接给答案

### 9.5 Session 复盘

每次 session 结束后生成复盘报告，至少包含：

- session 概要
- 状态时间轴
- 关键转折点
- 主要阻滞来源
- 可执行建议

### 9.6 知识谱系 / 路线图

首版作为 P1 能力，仅提供基础结构与局部可视化，用于说明：

- 当前任务涉及哪些知识点
- 当前进度位于何处
- 哪些节点可能是阻滞来源

## 十、数据模型

首版最小可用数据实体如下：

### 10.1 `Session`

记录一次完整学习会话。

关键字段建议：

- `session_id`
- `user_local_id`
- `task_id`
- `start_time`
- `end_time`
- `session_index`
- `split_reason`

### 10.2 `Task/Problem`

记录当前学习任务或题目元信息。

关键字段建议：

- `task_id`
- `task_type`
- `title`
- `difficulty_level`
- `knowledge_tags`
- `source_type`

### 10.3 `RawEvent`

记录最细粒度行为事件。

关键字段建议：

- `event_id`
- `session_id`
- `event_type`
- `event_time`
- `payload_summary`

### 10.4 `FeatureSnapshot`

记录窗口级特征快照。

关键字段建议：

- `snapshot_id`
- `session_id`
- `window_start`
- `window_end`
- `feature_values`

### 10.5 `StateVector`

记录状态向量与展示层状态输出。

关键字段建议：

- `state_id`
- `session_id`
- `snapshot_id`
- `dimension_scores`
- `top_states`
- `confidence`
- `evidence_summary`
- `generated_at`

### 10.6 数据保留策略

- `RawEvent` 默认保留 30 天
- 历史长周期存储优先保留 `FeatureSnapshot` 与 `StateVector`
- 支持自动清理与用户手动清空

## 十二、隐私、安全与 ESM 机制

### 12.1 隐私原则

- 行为数据必须只保存在本地
- 严禁上传原始代码与原始事件流
- 所有上传仅限统计摘要与抽象状态表示

### 12.2 凭证安全

- API Key 等敏感凭证优先存储在系统密钥链
- 如系统能力受限，退化为本地加密存储
- 禁止明文落盘

### 12.3 ESM（Encrypted State Machine）

LLM 侧只接收 `ESM`，即：

> 将本地学习行为压缩为去标识化的状态序列与统计摘要，并在传输前进行端到端加密；服务端仅基于该抽象表示进行分析，不接触原始代码与事件流。

### 12.4 ESM 抽象内容

服务端仅可见：

- 状态向量序列
- 聚合统计摘要
- 脱敏任务元信息

### 12.5 ESM 脱敏规则

- 时间离散化：按 `5s / 10s` 采样，去除精确时间指纹
- 数值分桶与裁剪：归一化并截断极值
- 类别泛化：语言、难度等仅保留枚举值
- 哈希标识：`session_id = HMAC(local_salt, raw_id)`，本地盐不外传

### 12.6 ESM 加密传输

- 数据体加密：`AES-256-GCM`
- 会话密钥封装：服务端公钥加密（`RSA/ECC Hybrid`）
- 传输链路：`HTTPS`
- 可选增强：证书校验 / pinning
- 重放保护：`nonce + timestamp`

### 12.7 数据可控性

系统提供：

- 一键清空本地数据
- 导出个人学习报告

## 十三、性能与非功能要求

### 13.1 性能指标

- 状态推理在本地完成，`P95 延迟 ≤ 100ms`
- 前端刷新频率：`2-3s`
- 单次 `session` 上限：`2 小时`
- 超过上限后自动切分 `session`

### 13.2 稳定性要求

- 状态输出需经平滑处理，避免过度抖动
- 实时反馈不应明显打断学习过程
- 关键路径不依赖持续联网

### 13.3 可用性要求

- 执行页保持轻量、低打扰
- 复盘页提供更完整说明
- 输出解释应短句化、因果化、可执行

## 十五、风险与权衡

### 15.1 状态推理准确性不足

**风险**：早期规则或轻量模型误判  
**影响**：错误干预，降低用户信任  
**对策**：

- 仅对高置信区间触发自动干预（如 `p > 0.7`）
- 其余只给建议，不强制打断
- 引入用户轻标注进行在线校准

### 15.2 多维状态空间过细

**风险**：维度多但缺少标注，难验证  
**影响**：看起来复杂，但可靠性不足  
**对策**：

- 首版只启用 `3-4` 个维度
- 其余维度保留为 latent
- 每个启用维度都要求明确计算式与可视化

### 15.3 隐私与云端能力的权衡

**风险**：只上传抽象状态会削弱云端分析能力  
**影响**：LLM 解释可能偏弱  
**对策**：

- 默认采用 `ESM`
- 可选增强模式（opt-in）上传更细粒度摘要，但仍不含代码
- LLM 提示词强调轨迹解释，而非代码级建议

### 15.4 理论强、工程时间紧

**风险**：理论模型实现不完整  
**影响**：答辩时容易被追问落地细节  
**对策**：

- 首版采用指数衰减 + 状态平滑近似
- 明确参数范围与更新频率
- 将完整模型放入演进路线，不作为首版硬承诺

### 15.5 通用学习与编程场景的落差

**风险**：抽象过高导致落地不清  
**影响**：评委质疑适用性  
**对策**：

- 首版严格绑定编程场景
- 所有特征与案例均来自编辑器行为
- 插件化仅做结构设计，不做跨域实现

### 15.6 冷启动与个体差异

**风险**：不同用户的行为习惯差异大  
**影响**：同一指标难以统一解释  
**对策**：

- 引入个人基线
- 前 `1-2` 个 session 作为校准期
- 使用相对变化量而非绝对阈值

### 15.7 实时性与稳定性的冲突

**风险**：刷新过快导致状态抖动  
**影响**：用户困惑  
**对策**：

- 前端 `2-3s` 刷新
- 使用状态平滑与滞后机制
- 展示 `TOP-2` 状态与置信度

### 15.8 可解释性与表达复杂度的冲突

**风险**：解释太复杂，用户看不懂  
**影响**：价值感下降  
**对策**：

- 前台只展示一句话因果与 `2-3` 个关键指标
- 详细轨迹与分析放在复盘页

### 15.9 数据增长与本地存储压力

**风险**：`RawEvent` 体量持续增长  
**影响**：本地性能与存储空间受损  
**对策**：

- 事件压缩与分桶
- 老数据优先只保留特征与状态
- 默认保留 `30 天` 并自动清理

### 15.10 评估与验证难

**风险**：缺少统一真值  
**影响**：难以证明系统有效  
**对策**：

- 设计轻量标注机制
- 结合自评与关键时刻问询
- 重点跟踪误触发率、有效干预率、停滞时长下降
