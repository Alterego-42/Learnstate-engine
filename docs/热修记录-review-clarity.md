# 热修记录：Review Clarity

## 背景
- 目标：把复盘页从“状态信息堆叠”改成“普通用户能快速读懂的学习复盘”。
- 范围：只处理 `ISSUE-004`、`ISSUE-005`，不改 report 后端结构，不引入复杂图表或额外基础设施。

## 本次改动
- 页面顺序改为“先结论、后证据、再建议”。
- 每个主要区块增加一句“这部分在看什么”。
- 吸收 `Process Narrator` 的最小补丁：
  - `REPORT_GENERATION_STEPS`
  - `reportProcessCaption`
  - 与这两部分直接相关的最小流程展示
- 状态与证据翻译成学习语言，例如：
  - `steady_flow` → `稳定推进`
  - `looping` → `反复回退`
  - `friction up` → `推进阻力较大`
  - `backtrack_loop_score up` → `出现来回回退`
- 时间轴节点改成学习阶段表达，例如“顺利推进阶段 / 卡住阶段 / 停下来整理阶段”。
- 建议压缩为 2-3 条短句，强调可立即执行。

## 明确放弃
- 不单独保留 `Process Narrator` 分支叙事。
- 不把流程区扩成独立主结构，不放到“先结论”前面。
- 不增加科学性展示层、更多解释卡片、额外状态页或后端依赖。
- 不改执行页、不改 daemon。

## 数据稀疏兜底
- 样本较少时提示“当前样本较少，结论主要基于已记录的少量状态”。
- 无明显转折时提示过程较平稳。
- 无明显卡点时提示整体更多在推进或探索。
- 证据缺失时提示主要根据状态轨迹整理。

## 页面前后差异
- 之前：用户需要自己从状态、转折、阻滞里拼凑“到底发生了什么”。
- 现在：页面顶部直接先回答“总体怎样 / 中间哪里卡住 / 最后怎么结束”。
- 之前：时间轴更像状态列表。
- 现在：时间轴更像学习阶段叙事。
- 之前：建议偏解释性。
- 现在：建议更短、更直接、更可执行。

## 现在这页如何回答两个问题
- 这页在说什么：
  - 先告诉用户这次 session 整体表现、关键卡点、结束方式；
  - 再用学习阶段时间线和关键转折解释过程；
  - 最后给下一步建议。
- 为什么它这么快生成，但不是随便生成：
  - 因为它直接读取已记录的行为事件、特征快照和状态点；
  - 再在前端即时聚合成时间线、转折点和建议；
  - 所以生成快，但结论仍然绑定真实 session 数据。
